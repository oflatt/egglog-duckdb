1//! # egglog-backend-trait
2//!
3//! A backend-agnostic interface to an egglog egraph. This crate exposes the
4//! [`Backend`] trait, along with companion traits ([`RuleBuilderOps`],
5//! [`BaseValuePool`], [`ContainerPool`]), so that the frontend `EGraph` in
6//! the top-level `egglog` crate can drive either the in-memory reference
7//! backend (`egglog_bridge::EGraph`) or the DuckDB-backed backend
8//! (`egglog_bridge_duckdb::EGraph`) through a single dyn-compatible API.
9//!
10//! ## Design principles
11//!
12//! - **Minimal IR changes.** This crate intentionally does NOT introduce a new
13//!   neutral rule IR. [`RuleBuilderOps`] mirrors `egglog_bridge::RuleBuilder`
14//!   one-for-one. The reference backend's `RuleBuilderOps` impl is a trivial
15//!   passthrough; the DuckDB backend's impl accumulates calls into its
16//!   internal data IR and submits them to the existing `compile_rule`
17//!   pipeline on `build()`. Frontend code (`BackendRule` in
18//!   `src/lib.rs::EGraph`) is unchanged in shape.
19//! - **Basic id and config types live here.** As of Phase 2 Commit 3,
20//!   `FunctionId`, `RuleId`, `ColumnTy`, `FunctionRow`, `FunctionConfig`,
21//!   `MergeFn`, `DefaultVal`, `QueryEntry`, `Variable`, and `VariableId` are
22//!   defined in this crate. `egglog-bridge` re-exports them so existing
23//!   callers continue to work. `Value`, `BaseValueId`, `ContainerValueId`,
24//!   `ExecutionState`, `ExternalFunction`, and `ExternalFunctionId` remain in
25//!   `egglog-core-relations` (already a neutral crate) and are re-exported
26//!   here for caller convenience.
27//! - **`Backend` is `Send + Sync` and dyn-compatible.** Methods that need
28//!   `T: BaseValue` or `C: ContainerValue` are factored onto
29//!   [`BaseValuePool`] and [`ContainerPool`], which expose Any-based dynamic
30//!   dispatch. A small set of generic helper functions (see the bottom of
31//!   this module) reintroduce the per-`T` sugar on top of those dyn traits.
32//! - **Cloning.** Backends must be cloneable via [`Backend::clone_boxed`].
33//!   The reference backend already derives `Clone`; DuckDB will need a
34//!   bespoke `clone_boxed` (e.g. database snapshot or replay buffer) when
35//!   it implements this trait.
36//!
37//! ## What is intentionally NOT in this trait
38//!
39//! - `with_execution_state`. The four callers in `src/` are migrated to
40//!   dedicated trait methods in a follow-up commit; this keeps `Backend`
41//!   dyn-compatible and avoids leaking the lifetime semantics of
42//!   `ExecutionState`.
43//! - `TableAction` / `UnionAction`. Under the minimal-change posture these
44//!   remain inherent methods on the bridge. The frontend keeps using them
45//!   directly. Backends that don't support them (DuckDB) error at the
46//!   primitive-registration call sites that touch them. The
47//!   [`Backend::supports_inline_table_lookups`] capability flag gates this.
48//! - A neutral rule IR. [`RuleBuilderOps`] is the seam instead.

112/// Proof-mode configuration for a UF-backed function (see
113/// [`Backend::add_uf_function`]).
114///
115/// When present, the UF function is backed by a provenance-tracking
116/// union-find: writes carry a per-edge proof column, and on each leader change
117/// the callback composes a proof that the displaced leader equals the new
118/// leader (reconstructed from the union-find's proof graph) and writes it into
119/// the onchange relation's trailing proof column. The composition interns
120/// `Trans` / `Sym` proof-constructor rows, so their backend [`FunctionId`]s are
121/// required here.
122#[derive(Copy, Clone)]
123pub struct UfProofConfig {
124    /// `Trans` proof constructor: `(Trans Proof Proof) -> Proof`.
125    pub trans: FunctionId,
126    /// `Sym` proof constructor: `(Sym Proof) -> Proof`.
127    pub sym: FunctionId,
128}

148/// How to resolve FD conflicts for a table.
149#[derive(Clone, Debug)]
150pub enum MergeFn {
151    /// Panic if the old and new values don't match.
152    AssertEq,
153    /// Use congruence to resolve FD conflicts: union the two colliding output
154    /// ids into the GLOBAL union-find (`$uf`). The standard congruence merge.
155    UnionId,
156    /// `--native-merge`: like [`MergeFn::UnionId`], but union the two colliding
157    /// output ids directly into the named per-sort UF-backed function's
158    /// union-find (its `@UF_Sf`) instead of the global `$uf`. Used by an FD-keyed
159    /// constructor view `(children) -> eclass` so the FD-conflict congruence
160    /// edge lands in exactly the union-find that owns the view's eclass column,
161    /// whose leader changes the view's relational rebuild then re-canonicalizes.
162    /// The `FunctionId` names that UF-backed function (the same association the
163    /// uniform [`Backend::register_native_merge_view`] contract records). Only
164    /// the native bridge resolves this variant; the dataflow/SQL backends route
165    /// the union via their own host-pass `native_merge_uf` association.
166    UnionIntoUf(FunctionId),
167    /// `--native-merge` WITHOUT `--native-uf` (the RELATIONAL union-find path):
168    /// like [`MergeFn::UnionIntoUf`], but the colliding output ids are written as
169    /// a union EDGE into the per-sort RELATIONAL parent table `@UF_S`
170    /// (`(S S) -> Unit :merge old`) — exactly the row the rule-encoded
171    /// `union()` helper writes: `(set (@UF_S larger smaller) ())`, where
172    /// `larger = max(cur, new)` and `smaller = min(cur, new)`. The merge returns
173    /// `min(cur, new)`. The relational maintenance rulesets
174    /// (singleparent / path_compress / uf_function_index) then propagate the edge
175    /// into `@UF_Sf`, and the view's relational `@rebuild_rule*` re-canonicalizes —
176    /// i.e. the rule-based rebuild absorbs the congruence cascade in batched
177    /// seminaive delta passes (no native-UF onchange re-scan blowup).
178    ///
179    /// Fields:
180    /// - `parent_table`: the relational `@UF_S` parent table to write the edge into.
181    /// - `unit`: the interned `Unit` value for the parent table's value column.
182    ///
183    /// Native bridge only (relational native-merge). The dataflow/SQL backends
184    /// keep their `--native-uf`-driven native-merge path.
185    UnionIntoParentTable {
186        parent_table: FunctionId,
187        unit: Value,
188    },
189    /// `--native-merge` PROOF mode on a SINGLE-OUTPUT backend (FlowLog/Feldera):
190    /// the proof-carrying, relational-UF counterpart of
191    /// [`MergeFn::UnionIntoParentTable`]. The 2-TABLE proof-congruence merge.
192    ///
193    /// It resolves the FD conflict of a single-output FD-keyed constructor view
194    /// `(children) -> eclass` whose per-row view proof (`eclass = f(children)`)
195    /// lives NOT in a second output column (the backend cannot host tuple output)
196    /// but in a separate single-output SIDE-TABLE `@<F>ViewProof :
197    /// (children..., eclass) -> proof`. On an FD conflict (same children, two
198    /// eclasses) it:
199    ///   1. reads each colliding row's view proof from `proof_side_table`, keyed
200    ///      `(children..., that_row_eclass)` — canon-at-creation guarantees a hit;
201    ///   2. composes the congruence edge proof `Trans(larger_pf, Sym(smaller_pf))`
202    ///      (same orientation as the rule-encoded `@congruence_rule`), where
203    ///      `larger = max(cur, new)`, `smaller = min(cur, new)`;
204    ///   3. writes the edge `[larger, smaller, edge_proof]` into the RELATIONAL
205    ///      proof-UF `@UF_S` (`parent_table`, `(S S) -> Proof :merge old`) — the
206    ///      same row the rule-encoded `union()` helper writes. The relational
207    ///      maintenance rules (singleparent / path_compress) then propagate it;
208    ///   4. returns the surviving (min) eclass.
209    ///
210    /// Fields:
211    /// - `parent_table`: the relational `@UF_S` proof-UF to write the edge into.
212    /// - `proof_side_table`: the `@<F>ViewProof` side-table to read view proofs from.
213    /// - `trans` / `sym`: the `Trans` / `Sym` proof constructors composing the edge.
214    ///
215    /// Only the FlowLog (and, later, Feldera) backends resolve this variant; the
216    /// native bridge has the A2 tuple proof view instead and treats it as
217    /// unsupported.
218    UnionIntoParentTableWithProof {
219        parent_table: FunctionId,
220        proof_side_table: FunctionId,
221        trans: FunctionId,
222        sym: FunctionId,
223    },
224    /// `--native-merge` in PROOF mode: the proof-carrying counterpart of
225    /// [`MergeFn::UnionIntoUf`]. Used by the `col0` (eclass) merge of a
226    /// tuple-output FD-keyed constructor view `(children) -> (eclass, proof)`.
227    ///
228    /// On an FD conflict (same children, two `(eclass, proof)` output rows) it
229    /// stages a proof-carrying congruence edge into the named proof-mode
230    /// per-sort `@UF_Sf` (a `DisplacedTableWithProvenance`, 4-column writes
231    /// `[lhs, rhs, proof, ts]`), then returns the surviving (min) eclass. The
232    /// staged edge proof is composed as `Trans(larger_proof, Sym(smaller_proof))`
233    /// over the two rows' proof columns, EXACTLY matching the orientation of the
234    /// rule-encoded `@congruence_rule` it replaces (which writes
235    /// `(set (@UF_Sf larger smaller) (Trans larger_pf (Sym smaller_pf)))`).
236    ///
237    /// Fields:
238    /// - `uf`: the proof-mode `@UF_Sf` UF-backed function to union into.
239    /// - `trans` / `sym`: the `Trans` / `Sym` proof constructors used to compose
240    ///   the edge proof.
241    /// - `eclass_col`: the value-column index of the eclass (this column's own
242    ///   index; the merge resolving `col0`).
243    /// - `proof_col`: the value-column index of the per-row term proof
244    ///   (`eclass = f(children)`).
245    ///
246    /// Only the native bridge resolves this variant; the dataflow/SQL backends
247    /// have no proof-mode native-UF and treat it as unsupported (the proof half
248    /// of native-merge is bridge-only).
249    UnionIntoUfWithProof {
250        uf: FunctionId,
251        trans: FunctionId,
252        sym: FunctionId,
253        eclass_col: usize,
254        proof_col: usize,
255    },
256    /// `--native-merge` in PROOF mode: the `col1` (proof) merge companion of
257    /// [`MergeFn::UnionIntoUfWithProof`]. Returns the term proof
258    /// (`eclass = f(children)`) of the SURVIVING (min) eclass — i.e. the proof of
259    /// whichever row has the smaller eclass value, matching the eclass kept by
260    /// the UF leader tie-break (`min`). This expresses the select-by-min that the
261    /// scalar `Old`/`New` leaves cannot.
262    ///
263    /// `eclass_col` / `proof_col` are the value-column indices of the eclass and
264    /// the proof, respectively. Bridge-only, like its companion.
265    EclassMinProof { eclass_col: usize, proof_col: usize },
266    /// The output of a merge is determined by applying the given ExternalFunction to the result
267    /// of the argument merge functions.
268    Primitive(ExternalFunctionId, Vec<MergeFn>),
269    /// The output of a merge is determined by looking up the value for the given function and the
270    /// given arguments in the egraph.
271    Function(FunctionId, Vec<MergeFn>),
272    /// Always return the old value for *this* value column (the column this merge resolves).
273    Old,
274    /// Always return the new value for *this* value column.
275    New,
276    /// The old value of value column `i`. Used by tuple-output (multi-value) merges, where a
277    /// column's merge may reference any output column of the OLD row. For a single-output
278    /// function, `OldCol(0)` is equivalent to [`MergeFn::Old`].
279    OldCol(usize),
280    /// The new value of value column `i`. The multi-value counterpart of [`MergeFn::New`].
281    NewCol(usize),
282    /// The value of KEY (input) column `i` (`cur[i]`). In an FD-conflict merge the
283    /// two rows share the same key, so the old and new key columns are equal —
284    /// hence a single `KeyCol(i)`. Lets a term-building custom `:merge` reference
285    /// the merge's input keys (e.g. to reconstruct the `(f keys merged)` view-term
286    /// proof). Bridge-only — the dataflow/SQL backends have no term-build proof
287    /// merge.
288    KeyCol(usize),
289    /// Always overwrite the new value for the given function with a constant. This is more useful
290    /// as a "base case" in a more complicated merge function (e.g. one that clamps a value between
291    /// 1 and 100) than it is as a standalone merge function.
292    Const(Value),
293    /// A merge for a tuple-output (multi-value) function: one [`MergeFn`] per value (return)
294    /// column. Each inner `MergeFn` produces that column's merged value and may reference any
295    /// output column via [`MergeFn::OldCol`] / [`MergeFn::NewCol`]. Its length determines the
296    /// number of value columns of the function (`n_vals`); `n_keys = schema.len() - n_vals`.
297    ///
298    /// `Columns` must appear only at the top level: a nested `Columns` (e.g. as an argument to a
299    /// [`MergeFn::Primitive`] or inside another `Columns`) is an error. A single-output function
300    /// uses a bare scalar merge (`Old`/`New`/`UnionId`/…), which is exactly a `Columns` of
301    /// length 1.
302    Columns(Vec<MergeFn>),
303    /// Insert a row into the given function's table (the `args` evaluate to the
304    /// full row), respecting that table's own merge. Returns the OLD value of the
305    /// resolving column (its return is meant to be discarded inside a
306    /// [`MergeFn::Seq`]). Models a `(set (f ...) v)` action inside a merge;
307    /// declares `f`'s table as a write-dependency so the side write is safe during
308    /// batched merges. Ported from PR #933 (`:merge`-multiple-actions). Bridge-only
309    /// — the dataflow/SQL backends treat it as unsupported.
310    TableInsert(FunctionId, Vec<MergeFn>),
311    /// Evaluate each merge function in order (for their effects) and return the
312    /// value of the last one. Models an action-block merge: the leading entries
313    /// are effects (e.g. [`MergeFn::TableInsert`] / [`MergeFn::Construct`]) and the
314    /// last is the value. Used to lower a term-building custom `:merge` body into a
315    /// single native merge. Ported from PR #933. Bridge-only.
316    Seq(Vec<MergeFn>),
317    /// Mint a pair-valued constructor inside a merge and return its output e-class.
318    /// `args` evaluate to the key (children); the first value column (the output) is
319    /// minted (the table's `FreshId` default) and the remaining value columns are
320    /// written from `value_args` (e.g. the term proof). Used to express a nested
321    /// constructor application inside a term-building custom `:merge` body. Ported
322    /// from PR #933 (FD proof encoding). Bridge-only.
323    Construct(FunctionId, Vec<MergeFn>, Vec<MergeFn>),
324    /// Conditional merge: evaluate `a` and `b`; if their values are EQUAL run and
325    /// return `then`, otherwise run and return `els`. Used to guard a term-building
326    /// custom `:merge` body (a [`MergeFn::Seq`]) so it only mints helper terms when
327    /// the function's old/new values genuinely differ — mirroring the rule-encoded
328    /// `@merge_rule`'s `(!= old new)` body guard. When `a == b` the rule-encoded
329    /// merge rule never fires, so no helper term is created; this variant reproduces
330    /// that exactly (return the already-equal value, no side effects). Bridge-only.
331    IfEq {
332        a: Box<MergeFn>,
333        b: Box<MergeFn>,
334        then: Box<MergeFn>,
335        els: Box<MergeFn>,
336    },
337}

393/// A backend that drives an egglog egraph.
394///
395/// Implementations: `egglog_bridge::EGraph` (reference, in-memory) and
396/// `egglog_bridge_duckdb::EGraph` (DuckDB-backed). The frontend `EGraph` in
397/// the `egglog` crate holds a `Box<dyn Backend>` and dispatches all state
398/// access through this trait.
399///
400/// ## Method correspondence
401///
402/// Each method's doc comment names the inherent method on
403/// `egglog_bridge::EGraph` that the bridge's `impl Backend` wraps. The
404/// reference backend's impl is meant to be one line per method
405/// (`fn foo(&self, ...) -> X { self.foo(...) }`). The DuckDB backend's
406/// translations are documented in `docs/backend_trait_design.md`.
407///
408/// ## Dyn-compatibility
409///
410/// All methods are object-safe. Methods that would otherwise be generic over
411/// `T: BaseValue` or `C: ContainerValue` live on the
412/// [`BaseValuePool`] / [`ContainerPool`] sub-traits, which use Any-based
413/// dynamic dispatch internally.
414///
415/// ## Bridge-only escape hatch via [`Backend::as_any`]
416///
417/// A handful of frontend constructs — most notably `TableAction::new` /
418/// `UnionAction::new` used inside `rust_rule` and `query` primitive
419/// `apply()` bodies in `src/prelude.rs` — depend on bridge-specific state
420/// (`egglog_bridge::EGraph`) that does not fit cleanly behind the trait
421/// surface. The Phase 1 design (see `docs/backend_trait_design.md`,
422/// "How `TableAction` and `UnionAction` are handled") explicitly keeps
423/// these types as bridge inherent methods rather than lifting them onto
424/// the trait.
425///
426/// To let callers reach the bridge-specific state when the backend field
427/// is `Box<dyn Backend>`, the trait exposes an `Any`-based downcast pair:
428///
429/// ```ignore
430/// // Inside a primitive that needs TableAction:
431/// let bridge = self
432///     .backend
433///     .as_any()
434///     .downcast_ref::<egglog_bridge::EGraph>()
435///     .expect("table actions are bridge-only");
436/// let action = egglog_bridge::TableAction::new(bridge, func_id);
437/// ```
438///
439/// The DuckDB backend's `as_any` returns a different concrete type
440/// (`egglog_bridge_duckdb::EGraph`); the downcast to
441/// `&egglog_bridge::EGraph` therefore fails on DuckDB. This is correct: the
442/// primitives that go through this path (`rust_rule` / `query` table-lookup
443/// helpers) are gated by [`Backend::supports_inline_table_lookups`] and
444/// are documented as unsupported on DuckDB in v1.
445///
446/// Using `&dyn Any` keeps this trait crate from having to name
447/// `egglog_bridge::EGraph` directly, which would re-introduce the
448/// `egglog-bridge ↔ egglog-backend-trait` dependency cycle that Phase 2
449/// Commit 3 deliberately broke.
450pub trait Backend: Send + Sync {
451    // -- table lifecycle ----------------------------------------------------
452
453    /// Register a function/relation/constructor and return its handle.
454    ///
455    /// Wraps `egglog_bridge::EGraph::add_table`.
456    fn add_table(&mut self, config: FunctionConfig) -> FunctionId;
457
458    /// Peek the [`FunctionId`] the next [`Backend::add_table`] call will return,
459    /// WITHOUT registering anything. Used for "knot-tying" a self-referential
460    /// `:merge`: the relational native-merge union-find function `@UF_Sf` needs
461    /// its OWN id while building its merge (a `MergeFn::TableInsert` into itself
462    /// for the recursive parent-union). The caller peeks the id immediately before
463    /// `add_table`.
464    ///
465    /// Bridge-only (this whole path is `--native-merge` on the native bridge). The
466    /// default panics: the dataflow/SQL backends never declare a self-referential
467    /// merge, matching the inert handling of the other bridge-only merge variants
468    /// (`Seq` / `TableInsert` / `IfEq`).
469    fn peek_next_function_id(&self) -> FunctionId {
470        panic!("peek_next_function_id is only supported on the native bridge backend")
471    }
472
473    /// `--nativerb` (native bridge, term-encoding native-UF, non-proof): register
474    /// a `@<F>View` view function (`view_func`) to be re-canonicalized by the
475    /// engine's native table rebuild against the per-sort `@UF_Sf` UF-backed
476    /// function (`uf_func`), replacing the encoding-level `@rebuild_rule*` rules.
477    /// The frontend calls this once per view function after all functions are
478    /// registered. Default no-op: only the native bridge supports a native table
479    /// rebuild; the dataflow/SQL backends drive their own host-pass rebuild.
480    fn register_nativerb_view(&mut self, _uf_func: FunctionId, _view_func: FunctionId) {}
481
482    /// `--native-merge` (term-encoding native-UF, non-proof): associate an
483    /// FD-keyed constructor view (`view_func`, keyed `(children) -> eclass`) with
484    /// the union-find (`uf_func`) that owns its eclass (OUTPUT) column. The view's
485    /// `MergeFn::UnionId` merge then injects a union into this UF on an FD conflict
486    /// (same children, two eclasses) — congruence done INLINE at insert time
487    /// instead of by a self-join rule. The frontend calls this before each run for
488    /// each not-yet-registered native-merge view. Default no-op: only backends
489    /// that inject the union on FD-conflict (currently FlowLog) implement it.
490    fn register_native_merge_view(&mut self, _uf_func: FunctionId, _view_func: FunctionId) {}
491
492    /// `--native-merge --proofs` on a `native_merge_views_coexist()` + single-output
493    /// backend (DuckDB): associate a constructor view (`view_func`, the all-columns
494    /// COEXIST shape `(children..., eclass) -> Proof`) with everything its
495    /// between-statement proof-congruence host pass needs. `parent_table` is the
496    /// RELATIONAL proof-UF `@UF_S` (`(S S) -> Proof :merge old`) the composed
497    /// congruence edge is written into; `proof_side_table` is the `@<F>ViewProof`
498    /// side-table `(children..., eclass) -> proof` the per-row view proofs are read
499    /// from; `trans` / `sym` are the `Trans` / `Sym` proof constructors used to
500    /// compose the congruence edge proof `Trans(larger_pf, Sym(smaller_pf))`.
501    ///
502    /// On an FD conflict (same children, two eclasses) the host pass self-joins the

993    ///
994    /// Ergonomic, generic-over-`R` wrapper around the object-safe
995    /// [`Backend::with_execution_state_dyn`]. The result is threaded back
996    /// out through a captured slot rather than `Box<dyn Any>`, so `R` does
997    /// not need to be `'static`. Mirrors
998    /// `egglog_bridge::EGraph::with_execution_state`.
999    pub fn with_execution_state<R>(&self, f: impl FnOnce(&mut ExecutionState<'_>) -> R) -> R {
1000        let mut f = Some(f);
1001        let mut out: Option<R> = None;
1002        self.with_execution_state_dyn(&mut |es| {
1003            let f = f.take().expect("with_execution_state closure called once");
1004            out = Some(f(es));
1005        });
1006        out.expect("with_execution_state_dyn must invoke its closure exactly once")
1007    }

1029// `RuleBuilderOps` — mirrors `egglog_bridge::RuleBuilder` one-for-one
1030// ---------------------------------------------------------------------------
1031
1032/// A lazily-computed failure message for RHS lookups / external calls.
1033///
1034/// The message is only built if the lookup or call actually fails at
1035/// runtime. Deferring it matters because the usual message form is
1036/// `format!("{span}: ...")`, and formatting a `Span` calls
1037/// `SrcFile::get_location`, which scans the source-file prefix. Building it
1038/// eagerly for every action in every rule is `O(rules * file)` — quadratic
1039/// when many generated rules (term/proof encoding) are compiled against a
1040/// large source file. Boxed so the trait stays dyn-compatible.
1041pub type PanicMsg = Box<dyn FnOnce() -> String + Send>;
1042
1043/// Operations on an in-progress rule.
1044///
1045/// This trait mirrors the public methods on
1046/// `egglog_bridge::RuleBuilder`. The bridge's impl is a trivial newtype
1047/// passthrough; the DuckDB impl accumulates calls into its internal
1048/// `duck::Rule` data IR and submits to `compile_rule` on
1049/// [`RuleBuilderOps::build`].
1050///
1051/// ## Variable creation
1052///
1053/// `new_var` and `new_var_named` allocate new variables. The returned
1054/// `QueryEntry` can be passed back to body atoms / actions. Each `QueryEntry`
1055/// carries its [`ColumnTy`] for runtime arity / type checking.
1056///
1057/// ## Unsupported on DuckDB
1058///
1059/// - [`RuleBuilderOps::subsume`][]: error.
1060/// - Complex `MergeFn::Function` / `MergeFn::Primitive` referenced via
1061///   `query_prim`: error at `build()` time when the rule references such a
1062///   merge.
1063pub trait RuleBuilderOps {
1064    /// Bind a new variable of the given type.
1065    ///
1066    /// Wraps `RuleBuilder::new_var`.
1067    fn new_var(&mut self, ty: ColumnTy) -> QueryEntry;
1068
1069    /// Bind a new variable of the given type with a display name (eases
1070    /// debugging).
1071    ///
1072    /// Wraps `RuleBuilder::new_var_named`.
1073    fn new_var_named(&mut self, ty: ColumnTy, name: &str) -> QueryEntry;
1074
1075    /// Add a table body atom. The final entry is the function's return
1076    /// value. When `is_subsumed` is `Some`, the atom is constrained to rows
1077    /// with that subsumption bit.
1078    ///
1079    /// Wraps `RuleBuilder::query_table`.
1080    fn query_table(
1081        &mut self,
1082        func: FunctionId,
1083        entries: &[QueryEntry],
1084        is_subsumed: Option<bool>,
1085    ) -> Result<()>;
1086
1087    /// Add a primitive body atom. The final entry is the return value.
1088    ///
1089    /// Wraps `RuleBuilder::query_prim`.
1090    fn query_prim(
1091        &mut self,
1092        func: ExternalFunctionId,
1093        entries: &[QueryEntry],
1094        ret_ty: ColumnTy,
1095    ) -> Result<()>;
1096
1097    /// Call an external function in the RHS, panicking with `panic_msg` on
1098    /// failure. Returns the result variable.
1099    ///
1100    /// Wraps `RuleBuilder::call_external_func`. `panic_msg` is a [`PanicMsg`]
1101    /// closure, evaluated only if the call actually fails.
1102    fn call_external_func(
1103        &mut self,
1104        func: ExternalFunctionId,
1105        args: &[QueryEntry],
1106        ret_ty: ColumnTy,
1107        panic_msg: PanicMsg,
1108    ) -> QueryEntry;
1109
1110    /// RHS: look up the value of `func(entries)`, with the function's
1111    /// configured default behavior on miss.
1112    ///
1113    /// Wraps `RuleBuilder::lookup`. `panic_msg` is a [`PanicMsg`] closure
1114    /// (evaluated only on miss when the function is `DefaultVal::Fail`).
1115    fn lookup(
1116        &mut self,
1117        func: FunctionId,
1118        entries: &[QueryEntry],
1119        panic_msg: PanicMsg,
1120    ) -> QueryEntry;
1121
1122    /// RHS: subsume the row keyed by `entries` in `func`.
1123    ///
1124    /// Wraps `RuleBuilder::subsume`.
1125    ///
1126    /// **DuckDB**: errors at this call site. Programs that need subsume
1127    /// must use the reference backend in v1.
1128    fn subsume(&mut self, func: FunctionId, entries: &[QueryEntry]) -> Result<()>;
1129
1130    /// RHS: set `func(entries[..n-1])` to `entries[n-1]`.
1131    ///
1132    /// Wraps `RuleBuilder::set`.
1133    fn set(&mut self, func: FunctionId, entries: &[QueryEntry]);
1134
1135    /// RHS: remove the row keyed by `entries` from `func`.
1136    ///
1137    /// Wraps `RuleBuilder::remove`.
1138    fn remove(&mut self, func: FunctionId, entries: &[QueryEntry]);
1139
1140    /// RHS: merge two values in the union-find.
1141    ///
1142    /// Wraps `RuleBuilder::union`.
1143    fn union(&mut self, l: QueryEntry, r: QueryEntry);
1144
1145    /// RHS: panic with the given message.
1146    ///
1147    /// Wraps `RuleBuilder::panic`.
1148    fn panic(&mut self, message: String);
1149
1150    /// Register a deferred-panic external function on the underlying egraph
1151    /// and return its id. Used (e.g. by `unstable-fn`) to bake a panic id
1152    /// into a wrapped function's call site.
1153    ///
1154    /// Wraps `RuleBuilder::new_panic`. Default panics for backends that do
1155    /// not support it; the bridge forwards to its `RuleBuilder`.
1156    fn new_panic(&mut self, _message: String) -> ExternalFunctionId {
1157        unimplemented!("new_panic is not supported on this backend")
1158    }
1159
1160    /// Skip tree-decomposition during query planning for this rule
1161    /// (the `:no-decomp` option / `--no-decomp`). Default no-op for
1162    /// backends that don't decompose (e.g. duckdb); the bridge
1163    /// forwards it to its `RuleBuilder`.
1164    fn set_no_decomp(&mut self, _no_decomp: bool) {}
1165
1166    /// Exclude the given function's table from being a seminaive delta focus
1167    /// for this rule: the rule never fires on *new* rows of this table joined
1168    /// against *old* rows of the others. Used by the term-encoding
1169    /// fast-rebuild (`--fast-rebuild` on the native bridge) to drop the
1170    /// always-empty `δview ⋈ uf_old` variant of the `view ⋈ @UF_Sf` rebuild
1171    /// rule (excludes the view table), keeping only `view ⋈ δuf`. Bit-exact
1172    /// with the full rebuild under canonicalize-at-creation.
1173    ///
1174    /// Default no-op: the dataflow/SQL backends implement their own
1175    /// fast-rebuild (`enable_fast_rebuild()`) and never receive these encoding
1176    /// rebuild rules in fast-rebuild mode; only the bridge forwards this to its
1177    /// `RuleBuilder`.
1178    fn set_focus_exclude_table(&mut self, _func: FunctionId) {}
1179
1180    /// Finalize the rule. Returns the registered [`RuleId`].
1181    ///
1182    /// Wraps `RuleBuilder::build`. The DuckDB impl hands the accumulated
1183    /// `duck::Rule` IR off to `compile_rule` and inserts it into the
1184    /// backend's rule registry. If any accumulated call referenced a
1185    /// feature the backend does not support (e.g. subsume on DuckDB), this
1186    /// is where the error surfaces.
1187    fn build(self: Box<Self>) -> Result<RuleId>;
1188
1189    /// Consume the builder and check whether its accumulated body
1190    /// matches at least one assignment in the current database.
1191    /// Actions accumulated in the builder are ignored. The bridge
1192    /// backend's default implementation panics; the DuckDB backend
1193    /// overrides this to compile the body to SQL and run it.
1194    /// Used by `(check …)` on the DuckDB path; the bridge keeps its
1195    /// existing external-func side-channel approach in `check_facts`.
1196    fn build_check(self: Box<Self>) -> Result<bool> {
1197        Err(anyhow::anyhow!(
1198            "build_check is not implemented for this backend"
1199        ))
1200    }
1201
1202    /// Rename an already-registered primitive on the backend so that
1203    /// the per-id name reflects the call-site's concrete type. Used
1204    /// to disambiguate built-in primitives whose egglog name is
1205    /// overloaded across sorts (e.g. `^` is XOR for i64 and POWER for
1206    /// f64; `+` for String is concat). The bridge backend doesn't
1207    /// need this (it dispatches via the typed `ExternalFunction`
1208    /// directly), so the default is a no-op. The duck rule-builder
1209    /// routes the call through the egraph it already holds mutably.
1210    fn rename_prim(&mut self, _id: ExternalFunctionId, _name: String) {}
1211}

1214// `BaseValuePool` — dyn-compatible base-value registry
1215// ---------------------------------------------------------------------------
1216
1217/// A registry for base-value types, exposed through dynamic dispatch so it
1218/// fits inside `dyn Backend`.
1219///
1220/// Generic-over-`T: BaseValue` helpers are provided at the bottom of this
1221/// module (`pool_get<T>`, `pool_unwrap<T>`, `pool_register_type<T>`,
1222/// `pool_get_ty<T>`); they wrap the dyn methods below.
1223///
1224/// The bridge's impl forwards to `egglog_core_relations::BaseValues`
1225/// directly. The DuckDB impl maintains its own `BaseValues`-shaped registry
1226/// in-process; entries for `i64`/`f64`/`bool`/`String`/`()` are encoded
1227/// inline into SQL columns where possible, and exotic types fall back to
1228/// an in-memory intern table identical to the bridge's.
1229pub trait BaseValuePool: Send + Sync {
1230    /// Register a new base-value type using its `TypeId` plus a factory
1231    /// that constructs a fresh typed intern table for the type.
1232    ///
1233    /// The factory is invoked at most once: if a type with the same
1234    /// `TypeId` is already registered, the existing [`BaseValueId`] is
1235    /// returned and the factory is dropped.
1236    ///
1237    /// Callers typically use [`pool_register_type`], the generic-over-`T`
1238    /// helper below, which constructs the appropriate factory and threads
1239    /// it through.
1240    ///
1241    /// Wraps `BaseValues::register_type_dyn` on the bridge side.
1242    fn register_type_dyn(
1243        &mut self,
1244        type_id: TypeId,
1245        factory: Box<dyn FnOnce() -> Box<dyn DynamicInternTable>>,
1246    ) -> BaseValueId;
1247
1248    /// Look up the `BaseValueId` for a registered base-value type by its
1249    /// Rust `TypeId`.
1250    ///
1251    /// Wraps `BaseValues::get_ty_by_id`.
1252    fn get_ty_by_type_id(&self, type_id: TypeId) -> BaseValueId;
1253
1254    /// Intern an opaque (already-boxed) base value, returning its `Value`
1255    /// handle. The `Box<dyn Any>` must contain a value of the type
1256    /// previously registered as `ty`.
1257    ///
1258    /// Wraps the dyn dispatch over `BaseValues::get<P>`. Concrete-`T`
1259    /// callers should prefer `pool_get<T>` below, which special-cases
1260    /// `T::MAY_UNBOX` and avoids boxing.
1261    fn intern_dyn(&self, ty: BaseValueId, value: Box<dyn Any + Send + Sync>) -> Value;
1262
1263    /// Extract a base value of the registered type `ty` from `val`. The
1264    /// returned `Box<dyn Any>` holds a value of the same type that was
1265    /// previously interned with `intern_dyn`.
1266    ///
1267    /// Wraps the dyn dispatch over `BaseValues::unwrap<P>`. Concrete-`T`
1268    /// callers should prefer `pool_unwrap<T>` below.
1269    fn unwrap_dyn(&self, ty: BaseValueId, val: Value) -> Box<dyn Any + Send + Sync>;
1270
1271    /// True iff a base-value type with the given `TypeId` is registered.
1272    fn has_ty(&self, type_id: TypeId) -> bool;
1273}
1274
1275/// Generic helper: register a `T: BaseValue` with the pool and return its
1276/// [`BaseValueId`]. Equivalent to `BaseValues::register_type::<T>`.
1277///
1278/// This sits outside the trait so the trait remains dyn-compatible. It
1279/// builds the typed intern-table factory closure that
1280/// [`BaseValuePool::register_type_dyn`] requires.
1281pub fn pool_register_type<T: BaseValue>(pool: &mut dyn BaseValuePool) -> BaseValueId {
1282    pool.register_type_dyn(
1283        TypeId::of::<T>(),
1284        Box::new(|| egglog_core_relations::new_intern_table::<T>()),
1285    )
1286}
1287
1288/// Generic helper: look up the `BaseValueId` for `T: BaseValue`.
1289pub fn pool_get_ty<T: BaseValue>(pool: &dyn BaseValuePool) -> BaseValueId {
1290    pool.get_ty_by_type_id(TypeId::of::<T>())
1291}
1292
1293/// Generic helper: intern a typed base value into the pool, returning its
1294/// `Value`. Mirrors `BaseValues::get<T>`. Honors `T::MAY_UNBOX` for
1295/// inline-encodable types.
1296pub fn pool_get<T: BaseValue>(pool: &dyn BaseValuePool, value: T) -> Value {
1297    if T::MAY_UNBOX
1298        && let Some(v) = value.try_box()
1299    {
1300        return v;
1301    }
1302    let ty = pool_get_ty::<T>(pool);
1303    pool.intern_dyn(ty, Box::new(value))
1304}
1305
1306/// Generic helper: extract a typed base value of `T` from a `Value`.
1307/// Mirrors `BaseValues::unwrap<T>`. Honors `T::MAY_UNBOX`.
1308///
1309/// # Panics
1310///
1311/// Panics if `val` does not correspond to a value of type `T` previously
1312/// interned in `pool` (matching the bridge's existing semantics).
1313pub fn pool_unwrap<T: BaseValue>(pool: &dyn BaseValuePool, val: Value) -> T {
1314    if T::MAY_UNBOX
1315        && let Some(p) = T::try_unbox(val)
1316    {
1317        return p;
1318    }
1319    let ty = pool_get_ty::<T>(pool);
1320    let boxed = pool.unwrap_dyn(ty, val);
1321    *boxed
1322        .downcast::<T>()
1323        .expect("BaseValuePool::unwrap_dyn returned wrong type")
1324}

1327// `ContainerPool` — stub-friendly container registry
1328// ---------------------------------------------------------------------------
1329
1330/// A registry for container values (`Vec` / `Set` / `Map` / `MultiSet` /
1331/// user-defined `ContainerValue` impls).
1332///
1333/// ## Two implementations
1334///
1335/// - **Reference backend**: delegates to
1336///   `egglog_core_relations::ContainerValues`. All methods succeed; the
1337///   pool participates in the EGraph's rebuild loop.
1338/// - **DuckDB backend**: an empty stub. All accessor methods return `None`
1339///   / empty iterators; all mutators return errors. This is safe because
1340///   the term-encoding gate (`program_supports_proofs` in
1341///   `src/proofs/proof_encoding_helpers.rs`) excludes every container-using
1342///   program from DuckDB's test combos. The stub's error path is a
1343///   defensive measure for programmer error, never a routine code path.
1344///
1345/// ## Why not just generic-over-`C`?
1346///
1347/// `register_val<C>` / `for_each<C>` are generic over `C: ContainerValue`,
1348/// which is incompatible with `dyn Backend`. We expose Any-based dynamic
1349/// dispatch here; concrete-`C` sugar is implemented as free functions in a
1350/// future commit (the call sites in `src/lib.rs:1924`, `serialize.rs`,
1351/// `extract.rs` will each adopt the dyn API directly because they already
1352/// thread a `TypeId` through).
1353pub trait ContainerPool: Send + Sync {
1354    /// True iff a container type with the given `TypeId` is registered.
1355    fn has_container_type(&self, type_id: TypeId) -> bool;
1356
1357    /// True iff this backend supports containers at all.
1358    ///
1359    /// Reference: `true`. DuckDB: `false`. Provided as a convenience so
1360    /// callers don't need to consult [`Backend::supports_containers`].
1361    fn enabled(&self) -> bool;
1362
1363    /// Look up the container value associated with `val`. Returns `None`
1364    /// when no entry is registered.
1365    ///
1366    /// The returned `Box<dyn Any>` holds an instance of the container's
1367    /// registered Rust type.
1368    ///
1369    /// Wraps `ContainerValues::get_val`. On DuckDB always returns `None`.
1370    fn get_dyn(&self, ty: TypeId, val: Value) -> Option<Box<dyn Any + Send + Sync>>;
1371
1372    /// Register a Rust container value, returning a fresh `Value` handle.
1373    ///
1374    /// On DuckDB this returns an error (containers unsupported).
1375    fn register_val_dyn(&mut self, ty: TypeId, value: Box<dyn Any + Send + Sync>) -> Result<Value>;
1376
1377    /// Iterate (id, container) pairs for all registered values of a
1378    /// container type. The callback receives the container as a
1379    /// `&dyn Any` of the registered concrete type.
1380    ///
1381    /// Wraps `ContainerValues::for_each<C>`. On DuckDB this is a no-op
1382    /// (the stub registry is empty).
1383    fn for_each_dyn(&self, ty: TypeId, f: &mut dyn FnMut(Value, &dyn Any));
1384
1385    /// Number of registered values of the given container type.
1386    fn size(&self, ty: TypeId) -> usize;
1387}
1388
1389/// Generic helper: register a `C: ContainerValue` with the pool, returning
1390/// its `Value`. On DuckDB this returns an error.
1391pub fn container_register_val<C: ContainerValue>(
1392    pool: &mut dyn ContainerPool,
1393    value: C,
1394) -> Result<Value> {
1395    pool.register_val_dyn(TypeId::of::<C>(), Box::new(value))
1396}
