# Changes

## [Unreleased] - ReleaseDate

- **`--feldera --native-uf` is now the PURE-ENGINE rebuild (the `view ⋈ @DispΔ` join runs ON the DBSP circuit), and `--feldera --native-uf --fast-rebuild` is the CUSTOM off-circuit host-pass rebuild — reframing the `+fastrb` axis for native-UF feldera as "custom/specialized rebuild vs pure-engine rebuild", both bit-exact.** Mirrors the flowlog reframe. Previously feldera's native-UF rebuild was ALWAYS the host pass (the `+nuf`/`+nuf+fastrb` distinction was only a dead host-side `δview ⋈ uf_old` probe). Now: under `--native-uf` alone, each `@rebuild_rule` is rewritten (`interpret::rebuild_rule_dd_ir`) by STRIPPING the always-empty `@UFChange_S` onchange body atom AND the impure `@canon_S` body guard prim, and appending a `@DispΔ(eqsort_col)` atom over a synthetic arity-1 displaced-ids relation (registered per UF in `add_uf_function`) — leaving a PURE RELATIONAL join `view ⋈ @DispΔ` the persistent DBSP engine accepts and routes through the fused circuit alongside the user/congruence rules. The constraint that forced this: feldera evaluates body prims ON-circuit and REJECTS impure ones, so the `@canon_S` find-or-self guard (host-only, never registered pure) would PANIC `plan_join` — it cannot be routed onto the circuit the way flowlog's DD host-prim-tail keeps it. Dropping the guard is sound: the head's `@canon_S` calls (run host-side in `apply_head`) re-canonicalize, and re-inserting an already-canonical row is a no-op, so the over-inclusive guard-free join is still bit-exact. `sync_displaced_relations` feeds each `@DispΔ` mirror — before the read snapshot — with the prior round's displaced ids (`native_uf_prev_displaced`, reset once per rebuild round at the drain boundary via `native_uf_rebuild_ran`), so the fused fed-diff drives the seminaive `view ⋈ δdisplaced`. The O(1) find stays in-core; the JOIN runs on DBSP. Under `--native-uf --fast-rebuild` the existing off-circuit reverse-index δuf host scan (`native_uf_rebuild_envs`) runs unchanged; the reverse-index + supporting state are maintained ONLY on that custom path (zero cost on the pure path). The pure-engine path pays DBSP's O(state) per-transaction integral, so it is SLOWER — that is the point: the `+fastrb` axis now quantifies feldera's host-pass win (math-microbenchmark `+nuf` pure vs `+nuf+fastrb` host: n7 0.1s vs 0.1s = 1.51×, n8 0.3s vs 0.2s = 1.51×, n9 0.8s vs 0.5s = 1.54×, n10 5.2s vs 3.7s = 1.42×, n11 55.6s vs 49.3s = 1.13× — the gap narrows at the largest size as both paths become rebuild-dominated), with identical (bit-exact) results. All four feldera cells (`--feldera`, `+fastrb`, `+nuf`, `+nuf+fastrb`) bit-exact `(print-size)` vs bridge-normal on integer_math, bool, delete, web-demo/path, merge-during-rebuild, complex-merge-func, merge-saturates, complex-merge-prim. (Feldera backend only; relational `--feldera`/`+fastrb` cells and the bridge/flowlog/duckdb paths unchanged. Supersedes the earlier feldera `+nuf` = dead-`δview`-probe entry below.)
- **The native bridge (default core-relations backend) now has all four term-encoding rebuild cells DISTINCT in perf yet bit-exact in result: `--term-encoding` (plain), `--term-encoding --fast-rebuild` (`+fastrb`), `--native-uf` (`+nuf`, full), `--native-uf --fast-rebuild` (`+nuf+fastrb`) — matching the duckdb/feldera/flowlog four-cell matrix.** The bridge had no `enable_fast_rebuild()` knob (it runs the encoding through the standard seminaive rule pipeline, not a host-pass rebuild), so the relational `+fastrb` was previously UNIMPLEMENTED and `--native-uf` alone was the term-DROPPED `view ⋈ δuf` (degenerate with `+nuf+fastrb`). Two encoding-level changes, gated on the new `EncodingState::fast_rebuild` (`EGraph::with_fast_rebuild`, wired from `--fast-rebuild` on the bridge CLI path only). (1) **Relational `+fastrb`:** the rebuild rule `view ⋈ @UF_Sf` has seminaive derivatives `view ⋈ δuf` (the real re-canonicalization) + `δview ⋈ uf_old` (empty under canonicalize-at-creation, but seminaive still scans δview). `--fast-rebuild` records the rule's view function so `EGraph::add_rule` calls the new `RuleBuilderOps::set_focus_exclude_table`, which excludes the view atom from being a seminaive delta focus in the bridge's `add_rules_from_cached` — dropping the `δview ⋈ uf_old` variant. The surviving `view ⋈ δuf` variant additionally drops the excluded view atom's `ts < mid_ts` lower bound so it joins δuf against the FULL view (old + just-inserted), mirroring duckdb's `t_view.ts < ?2` boundary — load-bearing for bit-exactness (otherwise a view row born and re-pointed in the same iteration is missed). (2) **`+nuf` full rebuild:** `rebuilding_rules_native_uf` now also emits a `δview ⋈ uf_old` probe rule (its only table atom is the view, so seminaive runs it on δview only; the `@canon_S` guard finds no stale row — empty result, mirroring the relational empty-term cost) under `--native-uf` WITHOUT `--fast-rebuild`; `--fast-rebuild` drops the probe, leaving the onchange-driven `view ⋈ δuf` (the prior `+nuf+fastrb` behaviour). `set_focus_exclude_table` is a no-op default on the dataflow/SQL backends (they implement their own fast-rebuild and never receive these encoding rules in fast-rebuild mode). Bit-exact `(print-size)` vs bridge-normal for all four cells on integer_math, bool, delete, web-demo/path, merge-during-rebuild, complex-merge-func, merge-saturates, complex-merge-prim; on math-microbenchmark `+fastrb` (7.34s) < `plain` (7.80s) and `+nuf+fastrb` (7.20s) < `+nuf` (7.53s) — confirming the δview probe genuinely runs in `+nuf`. Proof mode never enables fast-rebuild. (Native bridge only; relational/native-uf paths unchanged when the flag is off.)

The `__UF_CHANGED__` adaptive delta scan stays a SEPARATE mechanism engaged by `--native-uf` alone, so it covers all four combinations. Bit-exact `(print-size)` vs bridge-normal for all four cells on integer_math, bool, delete, web-demo/path, merge-during-rebuild, complex-merge-func, merge-saturates; `+nuf` (17.20s) is measurably slower than `+nuf+fastrb` (16.40s) on math-microbenchmark (the probe runs). (DuckDB backend only; relational `--duckdb`/`--duck-native-uf` paths unchanged.)
- **`--feldera --native-uf` alone is now the FULL native-UF rebuild (it runs the `δview ⋈ uf_old` probe), and `--feldera --native-uf --fast-rebuild` drops it — making all four feldera cells (`--feldera`, `+--fast-rebuild`, `+--native-uf`, `+both`) DISTINCT in perf yet bit-exact in result.** The rebuild join `view ⋈ uf` has two seminaive derivatives: `view ⋈ δuf` (the real work, the reverse-index displaced-id scan) and `δview ⋈ uf_old` (canonicalize the NEW view rows of this iteration against the pre-drain UF — always EMPTY under canonicalize-at-creation, but the relational rebuild still PAYS to probe each new row's find). Bit-exact `(print-size)` vs bridge-normal for all four cells on integer_math, bool, delete, web-demo/path, merge-during-rebuild, complex-merge-func; `+nuf` is measurably slower than `+nuf+fastrb` on math-microbenchmark (the probe runs). (Feldera backend only; relational `--feldera` and the bridge/flowlog/duckdb paths unchanged.)
- **`--native-uf` now means "the rebuild scopes to `view ⋈ δuf`" on every backend, by DEFAULT; `--fast-rebuild` now means the RELATIONAL empty-term drop (`δview ⋈ uf_old`).** The rebuild is the join `view ⋈ uf`; its incremental derivative is `δview ⋈ uf_old + view_all ⋈ δuf`. With canonicalize-at-creation the `δview ⋈ uf_old` term is always EMPTY. There are two distinct optimizations, now wired to the two flags: (A) NATIVE-UF provides the leader-change delta (the displaced set / `@UFChange_S`), so under `--native-uf` the rebuild ALWAYS scopes to `view ⋈ δuf` — the indexed onchange-delta rebuild on the bridge (`rebuilding_rules_native_uf`) and the reverse-index delta scan on the dataflow backends (feldera/flowlog `native_uf_rebuild_envs`, duckdb's `__UF_CHANGED__`-driven adaptive path), each keeping a full-scan fallback only for the first iteration / unseeded index. (B) `--fast-rebuild` (WITHOUT `--native-uf`) is the RELATIONAL fast-rebuild: it drops the always-empty `δview ⋈ uf_old` term in each dataflow backend's δuf-driven rebuild (the path previously gated only by `FELDERA_DELTA_REBUILD` / `DUCK_DELTA_REBUILD` / `FLOWLOG_DELTA_REBUILD`; the env vars still work as a fallback, OR'd with the flag). With `--native-uf` the flag is a no-op (the onchange/delta rebuild has no `δview ⋈ uf_old` term to drop). The encoding no longer consumes `--fast-rebuild` at all (removed `EncodingState::fast_rebuild`, `with_fast_rebuild`, `rebuilding_rules_native_uf_fullscan`); the flag is purely a backend-config knob (`config.fast_rebuild` → `enable_fast_rebuild()`). Bit-exact `(print-size)` vs bridge-normal across `--native-uf` on bridge/feldera/flowlog/duckdb and `--fast-rebuild` (relational) on feldera/flowlog/duckdb (integer_math, bool, delete, web-demo/path).

- **`--native-uf` (term mode) rebuild is now ≤ relational (`--term-encoding`): it was ~17–54% SLOWER; now ~7–24% FASTER, still bit-exact.** Two fixes to the onchange-driven rebuild in `proof_encoding.rs::rebuilding_rules_native_uf`. (1) **State-driven blowup:** the `@UFChange_S` onchange relation is append-only (one row per union, retained forever), so seminaive re-joined every later iteration's fresh — already-canonical — view rows against the *entire accumulated* onchange history (`δ@view ⋈ full@UFChange_S`), a super-linear cost (one mmb_9 iteration matched 40 546 rows vs ~12k total). A new drain rule (own `@uf_change_drain` ruleset, run after the rebuild *saturates* — it cannot interleave with `@rebuilding`, where congruence's leader-change callback writes the same relation, without corrupting its hash-cons index) deletes consumed onchange rows so the relation stays bounded across rebuild passes. (2) **Double-firing:** the rebuild emitted TWO rules per eq-sort view column (equi-joining the displaced id against each of `lhs_leader`/`rhs_leader`), the second of which always searched the join on the *surviving* leader only to be rejected by the guard. The leader-change callback now stores the displaced id `max(lhs_leader, rhs_leader)` as an explicit 6th onchange column (term mode only; user `:onchange` relations stay 5-column — the bridge callback picks 5/6 from the relation's declared arity), so a single indexed equi-join per column suffices — halving rebuild rules (62→31 on the math benchmark) and matches (2.94M→1.47M). A computed `(ordering-max ll rl)` join was tried and rejected: the planner can't index it, forcing a full onchange×view scan (30s+). Net: native-UF now drops the relational `@single_parent`/`@parent`/`@uf_function_index` maintenance AND has comparable rebuild firings, so it wins overall (mmb_9 0.13s vs 0.17s; math-microbenchmark run-11 6.4s vs 6.9s). Bit-exact vs `--term-encoding` (term + `--proofs`, `(print-size)` identical; combinators/naturals proof_testing pass). Proof-mode rebuild is untouched. (Native bridge, term mode; relational path unchanged.)

- **`--native-uf` is now PROOF-CAPABLE on the native bridge (`--native-uf --proofs`): proofs are composed in the union-find leader-change callback, keeping proof-mode native-UF fully native.** Ported `DisplacedTableWithProvenance` (petgraph proof-graph + `get_proof` shortest-path reconstruction) from upstream PR #782 into `core-relations`; `LeaderChange` gains a `write_proof` column. In proof mode `add_uf_function` backs the UF function with the provenance variant (`(S S) Proof` schema → 4-column writes `[a, b, proof, ts]`); on each leader change the table reconstructs the path `displaced → new_leader` itself (it can't read its own table back during its merge) and the bridge callback folds it into a `Proof` term — per-edge proofs `Sym`-wrapped for backward steps and chained with `Trans`, all minted via `TableAction::lookup_or_insert` — then records `(write_lhs write_rhs lhs_leader rhs_leader new_leader proof)` into the now-6-column onchange relation. The onchange/`Trans`/`Sym` tables are made read+write-deps of the UF function so they stay readable (and pre-buffered) during its merge. The encoding (`proof_encoding.rs`) reuses Wave 2's pair threading: `union()` passes the edge proof into `(set (@UF_Sf a b) proof)`; the onchange-driven rebuild threads the onchange proof column as the per-column congruence proof (`Congr` for child columns, `Trans(Sym …)` for a constructor's representative), single-stepping each displaced column to `new_leader` and chaining transitivity across firings — mirroring the relational `@UF_Sf` `pair-first`/`pair-second` rebuild. The CLI `--native-uf`+`--proofs` gate is lifted. RESULTS are bit-exact vs relational `--proofs` (integer_math, calc, bool `(print-size)` identical); proofs VALIDATE (`combinators`/`naturals` proof_testing pass under `--native-uf`, ~28–39 MB; the harness catches a forced-false check, confirming the runs are meaningful). Term-mode native-UF and the relational path are unchanged. (Native bridge only; clippy clean on the touched crates.)

- **Feldera backend: δuf-driven rebuild (`FELDERA_DELTA_REBUILD`, flag-gated) — ported from FlowLog; BIT-EXACT but performance-NEUTRAL on DBSP (the honest cross-backend finding).** The rebuild join `view ⋈ @UF_Sf` has incremental derivative `δview⋈uf + view⋈δuf`; because canonicalize-at-creation makes new view rows born canonical, the `δview⋈uf` term is empty and can be dropped. `FusedJoinImpl::step` now splits the rebuild ruleset's single DBSP `transaction()` into two ticks over the SAME (never-zeroed) integrals — the DBSP analog of FlowLog's two `drive_to`+`drain_into` epochs: (A) feed δview (non-transient), run a tick, keep only the non-rebuild (congruence) rules' output and DISCARD the rebuild rules' `δview⋈uf`; (B) feed δuf (the identity-on-miss `@UF_Sf` flat-index), run a second tick, keep the rebuild rules' `view⋈δuf`. Transient funcs are collected ONLY from `@rebuild_rule`-named rules' bodies (mirroring FlowLog's `rule_category=="canonicalize"` gate) — load-bearing, since a USER rewrite rule's instrumented body also reads `@UF_Sf`, so scanning every rule would wrongly split the user ruleset and drop its δview-driven matches. Bit-exact on math N=7/9/10 (1165/1330, 12067/11825, 70487/51682) AND the feldera files suite (981 pass / 6 fail — the pre-existing cykjson + f64_duckdb failures); default (flag-off, symmetric single-transaction) path unchanged. **Performance finding:** unlike FlowLog (~29% off canonicalize on raw DD, where timely's `step_while` has no fixed per-tick cost), this is performance-NEUTRAL-to-slightly-slower on DBSP (mmb_10 `step(transaction)` steady-state ~4.84s default vs ~4.96s delta-rebuild, within ±2% run-to-run variance): DBSP's `transaction()` re-evaluates its large fast-growing integrals every tick (the documented "circuit_step scales with INTEGRAL size, not delta/transaction count" property), so splitting the rebuild ruleset into two ticks adds ~a full extra integral-pass that cancels the savings from dropping the small `δview⋈uf` term. Kept behind the off-by-default flag as the negative-result record + the bit-exact two-substep construction. (Feldera-only; clippy clean.)

- **Canonicalize-at-creation (`EGGLOG_CANON_AT_CREATION`, flag-gated, flowlog-validated) + `DefaultVal::Identity`.** The term encoder, under the flag, replaces each eq-sort child `c` at constructor/view creation with an `(@UF_Sortf c)` lookup-or-self, tagging the rule `:unsafe-seminaive`, so a new row is born canonical w.r.t. the *current* (UF_old) union-find — `@UF_Sf` is the flat, rebuild-saturated id→root index, frozen during the user-rule phase. This makes the `δ(constructor) ⋈ UF_old` rebuild-join term semantically empty (new rows reference only UF_old-canonical ids), the precondition for a sound δUF-driven rebuild. The needed primitive — identity-on-miss find (miss ⇒ the id is its own leader) — is a new `DefaultVal::Identity` reusing `lookup_uf`'s `lookup_with_default` machinery pointed at `@UF_Sf` (the `@UF_*` tables are `Custom` functions whose real miss is `Fail`, so neither `Fail` nor `FreshId` was usable). Default OFF → encoding byte-for-byte unchanged for all backends (flag-off paths are no-ops). Implemented end-to-end for the FlowLog backend (`HeadOp::Lookup` evaluates the lookup-or-self host-side); flowlog files suite 62/62 with the flag ON, math-microbenchmark bit-exact N=7/9/10 (1165/1330, 12067/11825, 70487/51682). Other backends route `DefaultVal::Identity` to existing behavior and are not validated under the flag (flowlog-first). (Foundation for the δUF-driven custom rebuild; the DD dataflow restructure that actually drops the term is a follow-up.)

- **`:unsafe-seminaive`: widen typecheck primitive contexts (complete the option; sync with upstream PR #912).** Our independent `:unsafe-seminaive` implementation gated `check_lookup_actions` on the flag but still selected the primitive contexts (`Pure`/`Write` vs `Read`/`Full`) from plain `seminaive`, so function-table lookups in an `:unsafe-seminaive` RHS worked but read-primitives (`FullPrim`, e.g. `unstable-multiset-fill-index`) failed to resolve. Now `context_seminaive = seminaive && !unsafe_seminaive` drives context selection (evaluation still keys off `seminaive`), matching the backend's rule-build wiring, so an `:unsafe-seminaive` RHS admits both read-primitives and function-table lookups under seminaive (delta) evaluation. Adds `tests/unsafe-seminaive-read-prim.egg`. (Frontend only; no backend changes.)

- **FlowLog backend: fuse the per-rule DD dataflows into one shared timely worker per ruleset.** The DD path built one timely `Worker` PER RULE — math-microbenchmark spun up 79 workers / 173 InputSessions, redundantly arranging each shared body relation once per reading rule. `FusedDdJoin` now hosts ONE shared `Worker` with one `worker.dataflow` per ruleset (keyed by the sorted live atom-bearing rule-index list, mirroring Feldera's `FusedJoin`): each distinct body relation is ONE shared input `Collection`, each rule a left-deep join sub-stream with its own `inspect_batch` capture, clocked once per epoch (workers 79→6, sessions 173→66; never-clear/delta-fed incremental invariant + external epoch-drive preserved). The now-redundant input/output `.distinct()`s are dropped (fused input delta is already set-semantic; the env consumer reads only net-weight sign), restorable via `FLOWLOG_DD_KEEP_{INPUT,OUTPUT}_DISTINCT`. ## [0.1.0] - 2023-10-31

This is egglog's first release! Egglog is ready for use, but is still fairly experimental. Expect some significant changes in the future.

- Egglog is better than [egg](https://github.com/egraphs-good/egg) in many ways, including performance and new features.
- Egglog now includes cargo documentation for the language interface.

As of yet, the rust interface is not documented or well supported. We recommend using the language interface. Egglog also lacks proofs, a feature that egg has.
